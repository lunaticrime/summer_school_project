31. Périmètre et décisions d’architecture
Ce document décrit un premier MVP répondant au BPMN TO-BE du use case de 1 à 9 : réception du résultat, calcul des scores, 
détection des lacunes, diagnostic pédagogique, génération du plan, validation humaine, archivage, envoi à l’étudiant et 
notification de l’enseignant.
Un seul étudiant est traité par exécution.
Une matière et une évaluation par exécution.
Trois à cinq compétences par résultat.
Les calculs de score et de lacunes sont déterministes.
L’IA est utilisée uniquement pour le diagnostic, la génération et la régénération du plan.
Aucun plan n’est envoyé avant validation de l’enseignant.
Les tables n’utilisent aucune colonne JSON ou JSONB obligatoire.
Les tableaux skills et activities circulent entre workflows, puis sont insérés ligne par ligne avec un nœud Loop.
2. Architecture globale du MVP

  WF-02 Stockage du résultat

WF-02 - Stockage du résultat
Rôle : Enregistrer l’étudiant, l’évaluation, le résultat et une ligne par compétence.
Endpoint : POST /student-result/store
Tables utilisées : students, assessments, student_results, student_skill_results
Schéma du workflow
Webhook -> Prepare Student -> Upsert Student -> Prepare Assessment -> Upsert Assessment -> Upsert Student 
Result -> Prepare Skill Rows -> Loop Skills -> Insert Skill Row -> Count Stored Rows -> Verify Storage -> 
Respond
Nœuds Fusion AI
Ordre Nœud Fonction1 Webhook Trigger Recevoir le payload validé.2 Function prepare_student Construire les colonnes simples de students.3 Supabase Upsert students Upsert sur student_code.
4 Function prepare_assessmentConstruire les colonnes simples de assessments.5 Supabase Upsert assessments Upsert sur assessment_code.
6 Function prepare_resultConstruire student_results avec idempotency_key.7 Supabase Upsert student_results Upsert sur idempotency_key.8 Function prepare_skill_rows Ajouter student_result_id à chaque skill.9 Loop Itérer sur chaque compétence.10 Supabase Insert student_skill_results Insérer une ligne simple.11 Function count_inserted Compter les lignes réellement insérées.12 Function verify_storage Comparer received, prepared et stored.13 Respond to Webhook Retourner stored ou storage_incomplete.
Règles principales
Aucun tableau n’est inséré dans Supabase.
skills_received = skills_prepared = skills_stored obligatoire.
Un replay idempotent ne doit pas créer de doublons.
